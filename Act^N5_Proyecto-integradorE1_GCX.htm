Introducción

Durante las unidades se han estudiado los beneficios del uso e implementación de servicios web para la solución de problemas de comunicación entre otros sistemas. Si bien es cierto SOA es un patrón arquitectónico muy robusto y que ofrece granes beneficios, este ya viene en desuso para soluciones nuevas, es decir soluciones que parten desde 0. SOA Hoy en día se ve mas para colocar interfaces a otros sistemas, pero con la particularidad de que son sistemas existentes, muy robustos y típicamente montados en aplicaciones monolíticas. Ejemplo sistemas bancarios core donde no es posible desacoplar de primera mano una aplicación critica, por lo que se puede optar por exponer un servicio web del tipo jax-ws que sigue el patrón SOA.

La solución que se planteará al ser nueva será basada en microservicios utilizando el patrón arquitectónico REST FUL.

 

Desarrollo

 Actualmente muchos desarrolladores están dejando de utilizar arquitecturas SOA para pasarse a los microservicios, y muchas empresas líderes ya se han dado cuenta de que la arquitectura de microservicios es la mejor opción para sus organizaciones.

 

 

La arquitectura de microservicios es un método de desarrollo de aplicaciones software que funciona como un conjunto de pequeños servicios que se ejecutan de manera independiente y autónoma, proporcionando una funcionalidad de negocio completa.

Descripción de la organización

 

 

Gelée es una microempresa que nace en enero del año 2021.Su principal giro es la venta de gelatinas de diversos diseños, sabores y si el cliente lo desea un plus adicional que puede ser, globos de helio para cumpleaños, rosas a elegir o chocolates. Todo esto para brindar la mejor experiencia de servicios y hacer de una simple gelatina, un momento ideal para compartir en compañía de seres queridos o porque no, para el bonito detalle.

Lanzando su primera entrega, sale a producción en la ciudad de Iguala Guerrero México, donde el pedido #1 fue todo un éxito, cliente satisfecho en toda la extensión de la palabra. Si bien es cierto Gelée tiene una fuerte competencia en cuanto al giro de servicios y productos, se ha planteado destacar por ofrecer productos de excelente calidad y sabor a precios totalmente accesibles para todos.

Impacto de servicios web en organización

Gelée hoy en día las únicas plataformas digitales que emplea para darse a conocer como marca y realizar pedidos y desde luego mostrar la lista de productos es: Facebook y WhatsApp.

El impacto que se tendrá al iniciar con el desarrollo de soluciones basadas en servicios web, microservicios para este caso en particular, será el primer paso para poner a disposición interfaces REST que podrán ser consumidas tanto por un portal web, así como por aplicaciones móviles, es decir en este primer alcance Gelée pretende desarrollar y dejar disponible apis REST para consumo. Permitiendo así generar pedidos y consultar lista de productos vía rest. El siguiente paso será desarrollar una aplicación con el apoyo de xamarin para poder consumir estos servicios REST y así subirla a tiendas y usuarios ahora realicen pedidos y consultes productos de una manera mas personalizada desde sus dispositivos móviles.

Descripción de la situación/problema

Gelée a ganado una cartera de clientes a lo largo del año 2021, eso le ha llevado a tener un pool de pedidos diarios considerable, lo cual lleva a expandir la compañía y llegar a otros municipios de la región. Desde luego la matriz seguirá siendo Iguala, que como ya se menciono sigue concentrando un alto volumen de pedidos.

Al recibir más pedidos día a día, el administrar estos manualmente vía redes sociales y WhatsApp se está tornando una tarea demandante que absorbe mas tiempo del esperado. Y si esta tendencia se presentara en las siguientes sucursales, entonces es momento de subirse al mundo digital y solicitar un desarrollo que permita recibir pedidos, consultar productos de manera automática mediante el uso de APIS mediante servicios web (microservicios), esto potenciado por una app móvil Android.

Con una solución de este tipo, se podrá delegar tareas manuales a una plataforma que lo hará de manera automática y se podrá tener un mejor control e historial de pedidos que podrán realizar análisis estadístico y predictivo de mercado con la finalidad de poder tomar decisiones en momentos críticos o importantes.

 

Descripción de la metodología

Para el desarrollo de las primeras apis, para Gelée se plantea el uso de una metodología agile, en este caso Scrum.

¿Por qué scrum?

Uno de los principales beneficios que aporta scrum es el ritmo de trabajo sostenible, ya que se adapta a los cambios de manera inmediata consiguiendo un ritmo constante, tanto en duración del sprint como de esfuerzo. Scrum se centra en el empirismo y rápida sucesión de fallos a través de iteraciones (sprints), para obtener el mejor resultado posible y maximizar el valor entregado del producto.

Al emplear Scrum como marco de trabajo, se plantea ejecutar una serie de sprints donde el primer sprint será para un planing y gromming de las necesidades de Gelée, generación y priorización de backlog.

Como sprint #1 se iniciará con la generación de análisis y documentación de diseño a nivel arquitectónico y los swagger de las apis.

Este sprint será entrada para la iteración#3 donde se planea desarrollar y desplegar microservicios en ambiente DEV y realizar pruebas funcionales.

Si bien es cierto no se llegará a producción desde el primer sprint, si se estará haciendo entrega a producción de las apis rest de Gelée hasta llegar al consumo desde web y móvil.

El uso de servicios cloud de Amazon web services, ayudara a agilizar la entrega continua y disponer de servidores, contenedores y clúster de kubernetes donde estarán ejecutándose los microservicios. Lo cual será una gran ventaja para disponer de todo un ecosistema de apis y microservicios REST.

Justificación de arquitectura

Para la implementación de arquitectura, la propuesta de solución a nivel de plataforma se realiza empleando los servicios de Amazon web services: Api Gateway, Cloudwatch, ECS, EKS, load balancer, aws code pipeline, por solo citar algunos servicios a utilizar.

Se determina tomar las soluciones provistas por los siguientes puntos:

Facilidad de uso

AWS está diseñado para permitir que los proveedores de aplicaciones, los proveedores de software independientes y los distribuidores puedan hospedar de una forma rápida y segura su aplicación, tanto si es una aplicación existente como si es una nueva aplicación basada en SaaS. Puede utilizar la consola de administración de AWS o las API de servicio web bien documentadas para obtener acceso a la plataforma de hospedaje de aplicaciones de AWS.

Flexible

AWS le permite seleccionar el sistema operativo, el lenguaje de programación, la plataforma de aplicaciones web, la base de datos, así como el resto de los servicios que se necesite. Con AWS, se tendrá acceso a un entorno virtual que le permite cargar el software y los servicios que necesita una aplicación. Esto facilita el proceso de migración de las aplicaciones existentes y mantiene las opciones para crear nuevas soluciones.

Rentable

Únicamente se tendrá que afrontar el costo de la potencia de cómputo, el almacenamiento y demás tipos de recursos que se utilice, sin contratos a largo plazo ni compromisos iniciales.

Escalabilidad y alto desempeño

Con las herramientas de AWS, Auto Scaling y Elastic Load Balancing, una aplicación podrá ampliarse o reducirse según la demanda. Gracias al respaldo de la sólida infraestructura de Amazon.

Seguro

Aplica un enfoque integral para proteger y reforzar nuestra infraestructura, incluidas medidas físicas, operativas y de software.

 

En la imagen previa, se muestra un alto nivel de la solución basada en AWS, donde al tener expuestos microservicios, podrán consumirse desde portales web, dispositivos móviles entre otros que tengan la instrumentación para consumo de apis rest. Una vez dentro de la subnet de aws, el api Gateway recibí el request y si este es válido lo dejará pasar a la subnet privada donde estará corriendo eks con los contenedores que tienen corriendo los endpoint de Tomcat. Los logs y trazas serán enviadas a aws cloudwatch, permitiendo tener trazabilidad de peticiones REST.

 

 

A un nivel completo, adicional a los componentes descritos del flujo que soportara el api rest de Gelée. El equipo de desarrollo para poder pasar entre ambientes ejecutara un pipe line de aws. Donde desde git se dispara el job para un code commit de Amazon, este corre los pasos de compilación, test y build. Se construye la imagen de Docker y se manda al registry o ECR para dejarla en el repositorio y pueda ser desplegada a los pods de EKS.

Java Microservicios

En la imagen previa se muestra la propuesta tecnología que se pretende cubrir con la solución completa, como ya se indicó en la primera etapa del desarrollo será la entrega del api rest con los microservicios corriendo, el equipo de desarrollo hará su demo desde ambientes locales, esto atiende a no generar costos en ambientes cloud. Se mantiene en análisis el uso de travis o sustituir por Jenkins, pero de momento se deja ese stack propuesto.

De cara a desarrollo java, el framework principal será el uso de spring boot y la suite de spring para acelerar el desarrollo de los microservicios.

 

Conclusiones

 El uso de arquitecturas orientadas a servicios o el uso de arquitecturas orientadas a microservicios ofrece grandes ventajas a aplicaciones existentes o desarrollo de aplicaciones totalmente nuevas. Donde para SOA permite exponer funcionalidad mediante una interfaz hacia otros sistemas clientes. De lado de microservicios, permite crear soluciones ligeras y a la vez potentes, utilizan una arquitectura REST Ful y son servicios independientes de otros servicios.

Se observa que para cual sea de ambas variantes arquitectónicas pueden existir soluciones que aceleran la adopción e implementación de servicios y microservicios. Acortando tiempos de implementación y desarrollo. Al final será elección de cada compañía y los análisis de sus equipos de TI y arquitectos. AWS es un proveedor muy interesante para adoptar soluciones basadas en nube.
